# Ph1. 企画フェーズ指示書

## フェーズ概要

| 項目 | 内容 |
|------|------|
| 目的 | プロジェクトの方向性を定める（大きな絵を描く） |
| 担当 | AG |
| 成果物 | 企画書（project_overview.md） |
| 次フェーズ | Ph2. 要件定義 |

---

## ワークフロー

```
Step1: ヒアリング開始
  ↓
Step2: 目的（Why）を聞く
  ↓
Step3: ゴール（What to Achieve）を聞く
  ↓
Step4: 出力（Output）を聞く
  ↓
Step5: 入力（Input）を聞く
  ↓
Step6: ターゲットを確認
  ↓
Step7: 曖昧さチェック
  ↓
Step8: リスクチェック
  ↓

  ↓

```

---

## Step1: ヒアリング開始

施主に以下を伝えてヒアリングを開始する。

```
企画フェーズを始めるよ。
このフェーズでは「どんなサービスを作りたいか」の大きな絵を描いていくね。
詳細な仕様は次のフェーズで詰めるから、まずは夢を語ってもらう感じで大丈夫。

いくつか質問していくから、思いつくままに答えてね。
```

---

## Step2: 目的（Why）を聞く

**聞くこと**: なぜこのサービス/アプリを作りたいのか

**質問例**:
- このサービスを作ろうと思ったきっかけは？
- 誰のどんな課題を解決したい？
- 既存のサービスではなぜダメ？

**深掘りのヒント**:
- 「自分が困っている」のか「誰かのために作りたい」のかを明確にする
- 課題が曖昧な場合は具体的なシーンを聞く（「いつ、どこで、どんなときに困る？」）

---

## Step3: ゴール（What to Achieve）を聞く

**聞くこと**: このサービスで何を達成したいのか

**質問例**:
- このサービスが完成したら、どんな状態になっていてほしい？
- 成功の定義は？（ユーザー数？売上？自分の満足？）
- 最低限これだけは実現したい、というコア機能は？

**深掘りのヒント**:
- 「あれもこれも」になりがちなので、優先順位を意識させる
- 「もし1つだけ機能を選ぶなら？」という質問が有効

---

## Step4: 出力（Output）を聞く

**聞くこと**: ユーザーはこのサービスから何を得るのか

**質問例**:
- ユーザーがこのサービスを使った結果、何が手に入る？
- どんな画面や情報を見せたい？
- ユーザーが「使ってよかった」と思う瞬間は？

**深掘りのヒント**:
- 具体的なアウトプットのイメージを言語化させる
- 「〇〇が一覧で見える」「△△がダウンロードできる」など

---

## Step5: 入力（Input）を聞く

**聞くこと**: サービスを動かすために何が必要か

**質問例**:
- ユーザーは何を入力/登録する？
- データはどこから来る？（ユーザー入力？外部API？管理者が登録？）
- 初期データは必要？

**深掘りのヒント**:
- 入力の手間が大きすぎると使われないサービスになる
- 「本当にユーザーに入力させる必要がある？」を問う

---

## Step6: ターゲットを確認

**聞くこと**: 誰がいつどこで使うのか

**質問例**:
- 想定しているユーザーは？
- どんなシーンで使う？（通勤中？仕事中？休日？）
- PCメイン？スマホメイン？

---

## Step7: 曖昧さチェック

以下に該当する場合、**次に進まず深掘りする**。

| チェック項目 | NGの例 |
|-------------|--------|
| 目的が不明確 | 「なんとなく面白そうだから」「流行ってるから」 |
| ターゲットが曖昧 | 「みんなに使ってほしい」「誰でも」 |
| ゴールが測定不能 | 「便利なものを作りたい」「いい感じにしたい」 |
| コア機能が決まっていない | 「全部大事」「優先順位はない」 |
| 出力イメージがゼロ | 「とりあえず作ってみて」 |

**待ったのかけ方**:
```
企画を進める前に、もう少し詰めたい部分があるよ。
〇〇が曖昧だと、要件定義で手戻りが発生しやすくなるんだ。
もう少しだけ深掘りさせてもらっていい？
```

---

## Step8: リスクチェック

以下の観点でリスクを確認し、問題があれば施主に伝える。

### 法的・倫理的リスク（即アウト or 要確認）

| チェック項目 | 具体例 |
|-------------|--------|
| 著作権・知的財産権 | 他者のコンテンツを無断で利用していないか |
| 個人情報・プライバシー | 個人情報を不適切に収集・利用していないか |
| 利用規約・API規約 | 外部サービスの規約に違反していないか |
| 公序良俗 | 社会的に問題のあるサービスではないか |
| 業法・許認可 | 特定の許認可が必要な事業領域ではないか（金融、医療など） |

**判定**:
- 明らかにアウト → 企画を中止、または方向転換を提案
- グレー → 施主に確認を促す（「〇〇について調べた方がいいかも」）

### 技術的リスク（要注意フラグ）

| チェック項目 | 具体例 |
|-------------|--------|
| 実現可能性 | 現在の技術で実現可能か |
| 個人開発の範囲 | 一人で扱える技術領域か |
| 外部依存 | 特定のAPIやサービスに強く依存していないか |
| 専門知識 | 特殊な専門知識（機械学習、暗号など）が必要か |

**判定**:
- 高リスク → 「この部分は技術的に難しいかも」と警告
- 詳細は Ph3（技術選定）で検討

### コスト的リスク（要注意フラグ）

| チェック項目 | 具体例 |
|-------------|--------|
| 初期コスト | 有料サービス・ツールが必須か |
| 継続コスト | 月額費用がかかるサービスに依存するか |
| スケール時のコスト | ユーザーが増えたとき費用が跳ね上がらないか |
| 無料枠の範囲 | 無料枠で収まる見込みはあるか |

**判定**:
- 高リスク → 「コスト面で注意が必要」と警告
- 詳細は Ph3（技術選定）で検討

**リスクがある場合の伝え方**:
```
この企画、〇〇の観点でリスクがありそうだよ。
今すぐダメってわけじゃないけど、次のフェーズで詳しく検討した方がいいね。
場合によっては代替案を考えることになるかも。それでも進める？
```

---

## Step9: 企画書作成・施主承認

ヒアリング内容をもとに、企画書テンプレート（`ph-1-planning.tmp.md`）を埋めて施主に提示する。

```
ここまでのヒアリング内容を企画書にまとめたよ。
内容を確認して、問題なければ「OK」と言ってね。
修正したい部分があれば教えて。
```

---

## 完了条件

以下のすべてを満たしたら、企画フェーズ完了として Ph2 へ進む。

- [ ] 曖昧さチェックをパスしている
- [ ] リスクチェックを実施している（法的リスクがあれば解消済み、技術・コストリスクは認識済み）
- [ ] **企画書（project_overview.md）が作成されている**
- [ ] 施主が「これでOK」と言っている

**次のセッションへの引き継ぎ**:
企画フェーズ完了時点で企画書を保存し、Ph2開始時に参照できるようにしておく。

## AGの姿勢

- この段階では「全部は叶わない」ことを前提に、夢を語ってもらう
- 否定せず、でも現実的なフィードバックは適宜入れる
- 「それいいね！ただ、〇〇という観点もあるけどどう思う？」というスタンス
- 詳細な仕様や技術的な話は、このフェーズでは深入りしない
